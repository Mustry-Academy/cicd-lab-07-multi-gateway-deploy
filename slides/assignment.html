The assignment · ground rules
READ FIRST
How this works: you are contributors now, and the repo is locked down
  • We're here to get reps in. Everything from the past four days, on repeat: feature branch, commits, PR, tag, deploy.
  • No fork this time. Accept the contributor invite for git@github.com:Mustry-Academy/cicd-lab-07-multi-gateway-deploy.git, clone it, and push your branches straight to our repo.
  • We locked the repo down, so no funny business. Nothing reaches main without a pull request, green checks and an approval. Yes, we tried pushing to main. It says no, even to us.
  • Only release.yaml decides what gets deployed. And it deploys straight to production: no dev, no test, no net. That's why the lock is on the door.
  • Sam or Jasper approves every PR. Two reviewers, five of you: there will be a queue. Small, tidy PRs jump it.
  • You deploy to site 7. The deploy workflow runs on that site's runner: runs-on: [self-hosted, oatmakers-site-7]. Already set up; you never touch the workflows.
  • You get admin on our real online gateway: https://cloud.mustrysolutions.com, login admin / MergeIntoMain!. Have a look around before you start, and please be gentle with it: everyone's work lands on this one gateway.
Ask early

Stuck for more than a couple of minutes? Say so in the room. Don't sit blocked in silence; that's exactly what we're here for.

The assignment · the mental model
MODEL
One repository, two jobs: the projects and the release file live together
Keep this picture in your head for the whole hour. Every move you make is on one side of it, and the seam between the two sides is still a version string.
projects/ + config
"What have we built?"
  • Everything the gateway runs lives here: the projects and the config, which includes the Ignition tags.
  • A git tag like wout@v1.0.0 marks a version: a point in history you can deploy. Nothing gets compiled; the deploy ships the files as they are at that tag.
  • A tag that was never pushed is a release that does not exist.
release.yaml
"What runs on production, right now?"
  • One file, one production gateway: site 7, the desired state.
  • You change a version number, and a deploy happens.
  • Nothing deploys any other way: the file is the only trigger.
One repo, by choice

In the teaching you saw option B: split the infra part (release files, gateway config) into its own repo, owned by another team. Both are valid. One repo keeps this lab small, and the seam between the two jobs is still a version string.

Why this order

Part 1 proves the whole pipeline to you alone: PR, review, tag, release, live. Part 2 puts five people on one project and one version stream, which is where coordination starts to hurt. Part 3 is what real teams do all day: build on work that is still landing.

Part 1 · your project · A
1A
Create the project with your name and get it through review
Clone the shared repo, add a project named after yourself, and take it through the same review gate everything else goes through.
1
Accept the invite and clone. No fork: your branches go straight to the shared repo.
2
Branch and add your project: projects/<yourname>/ with one simple Perspective view that shows your name. The exercise file points at a skeleton you can copy.
3
Push and open the PR. Watch the checks, then wait for Sam or Jasper to approve. If you get review comments, address them and push again: same PR.
4
Merge once it's green and approved.
git clone git@github.com:Mustry-Academy/cicd-lab-07-multi-gateway-deploy.git
cd cicd-lab-07-multi-gateway-deploy
git switch -c feature/wout-project
# add projects/wout/ with your welcome view …
git add projects/wout/
git commit -m "feat: add wout project with welcome view"
git push -u origin feature/wout-project
Observe

Only your project's checks run. Path-filtered CI: five people can open five PRs and nobody waits on anyone else's tests.

Observe

Merging deployed nothing. Main means "built and could ship", not "shipped". Your project is not on production yet.

Part 1 · your project · B
1B
Tag your release and add your project to release.yaml to deploy it
Two moves: make the release exist, then point production at it. In that order, or the deploy has nothing to ship.
1
Tag it: git switch main → git pull → git tag <yourname>@v1.0.0 → push the tag. The tag is the release: a named version of your project that release.yaml can point at. Now the release exists.
2
Pin it: new branch, add one line to release.yaml: your project at v1.0.0. PR, approval, merge.
3
Watch the deploy action pick the job up on the site 7 runner and ship your project to production.
4
Verify live: open https://cloud.mustrysolutions.com, log in with admin / MergeIntoMain!, find your view.
# release.yaml
 gateway: oatmakers-site-7
 projects:
   oatmakers:        v2.0.0
+  wout:             v1.0.0
Observe

The deploy checks out your tag and copies your project's files to the gateway. Nothing gets built. And that is why the tag came first: release.yaml can only point at versions that exist.

Observe

The runner label routes the deploy: runs-on: [self-hosted, oatmakers-site-7]. Same repo, but only site 7's runner picks this job up. More sites would mean more labels, not more pipelines.

Your project, your tag, your line in release.yaml, your view live on production. You just did the full loop every deploy in this lab will use.

The assignments
Tom lays the data foundation; Gregory builds the dashboard on top.
TOM
Set up the database connection, a migration with a seed, and a view on the data
  • Create the DB connection using the referenced secret POSTGRES_USERNAME. The secret is already on the environment: you reference it, you never see the value. Lab 06's secrets ladder, for real.

  • Enable tag history on Wout's tags, storing into Tom's database connection.

Part 3 · extra · build on each other
3 · EXTRA
Use each other's work: combine the tags, the database, the charts and the functions
Everything from Part 2 is now shared infrastructure. Build on it, fast, together.
1
Cross the challenges: store more tags in the database, add to the migration scripts, put an Embr chart on Tom's data, call Stephan's function from your view. Anything goes, as long as it ships by PR.
2
Keep PRs small and fast. One small change that merges beats a big one that sits in review while main moves under it.
3
Stay current: pull main often and rebase your branch. Most merge conflicts die before they are born.
4
Solve the conflicts that do happen. With five people in one project they will. That is not the pipeline failing you; that is the pipeline making the collision visible before production.
# the rhythm of part 3
git switch main && git pull
git switch -c feature/chart-on-downtime-data
# small change, small commit …
git push -u origin feature/chart-on-downtime-data
# PR → review → merge → next free tag → release.yaml
No script

This part has no steps to follow. Work together, build fast, get through the PRs. The team that keeps its PRs small will ship the most.
