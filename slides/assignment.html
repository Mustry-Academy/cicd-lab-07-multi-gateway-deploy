The assignment · ground rules
READ FIRST
How this works: you are contributors now, and the repo is locked down
  • We're here to get reps in. Everything from the past four days, on repeat: feature branch, commits, PR, tag, deploy.
  • No fork this time. Accept the contributor invite for git@github.com:Mustry-Academy/cicd-lab-07-multi-gateway-deploy.git, clone it, and push your branches straight to our repo.
  • We locked the repo down, so no funny business. Nothing reaches main without a pull request, green checks and an approval. Yes, we tried pushing to main. It says no, even to us.
  • Two deploy channels, two triggers. Every push to main lands on your own test gateway automatically (you set that up in Part 1). Only a change to release.yaml reaches the shared production gateway, and that's the one behind the lock.
  • Sam or Jasper approves every PR. Two reviewers, five of you: there will be a queue. Small, tidy PRs jump it.
  • You deploy to site 7. The deploy workflow runs on that site's runner: runs-on: [self-hosted, cicd-capstone]. Already set up; you never touch the workflows.
  • You get admin on our real online gateway: https://cloud.mustrysolutions.com, login admin / MergeIntoMain!. Have a look around before you start, and please be gentle with it: everyone's work lands on this one gateway.
Ask early

Stuck for more than a couple of minutes? Say so in the room. Don't sit blocked in silence; that's exactly what we're here for.

Part 1 · your personal test gateway
1
Set up a test gateway that auto-deploys every merge to main
Before you deploy anything to production, give yourself somewhere safe to see it first: a second local gateway in dev deployment mode that a workflow of your own updates on every push to main. Do this once, at the start, then leave it running all afternoon.
1
Give the test gateway its own clone: git clone … ../lab07-test, next to your working clone. The pipeline owns that clone; yours never gets yanked around mid-branch.
2
Start the provided test pair: the repo already carries your test gateway (port 8090, dev config mode) and your laptop runner behind one compose profile: LAB_USER=<yourname> RUNNER_TOKEN=<from Sam or Jasper> docker compose --profile test up -d. The runner registers with the label [self-hosted, <yourname>-local]; only jobs asking for your label run on your machine.
3
Author your workflow: .github/workflows/test-<yourname>.yml triggers on push: [main], runs on your label, fast-forwards the test clone (mounted at /workspace/lab07-test in your runner) and restarts lab07-test-gateway so it boots the fresh files — a gateway only reads externally-changed files at boot or on an authenticated scan.
4
PR it in like anything else. Branch, PR, green checks, approval, merge. The only file you touched is your own workflow.
# .github/workflows/test-<yourname>.yml
name: test-<yourname>
on: { push: { branches: [main] } }
jobs:
  deploy:
    runs-on: [self-hosted, <yourname>-local]
    steps:
Observe

Your workflow only runs on your machine. The runner label is the whole routing story: five people can each have a push: [main] workflow and each one deploys only to its owner's test gateway.

✅ Verify it works

Open a small PR that changes a dashboard view. Once it merges to main, watch your test-<yourname> workflow run on your runner and the change appear on localhost:8090 on its own. That is your test gateway tracking main for you, hands-free, before anything is ever tagged for production.

Part 2 · your project · A
2A
Create the project with your name and get it through review
Clone the shared repo, add a project named after yourself, and take it through the same review gate everything else goes through.
1
Accept the invite and clone. No fork: your branches go straight to the shared repo.
2
Branch and add your project: projects/<yourname>/ with one simple Perspective view that shows your name.
3
Push and open the PR. Watch the checks, then wait for Sam or Jasper to approve. If you get review comments, address them and push again: same PR.
4
Merge once it's green and approved. Then open localhost:8090: your Part 1 workflow has already put your new project on your test gateway, hands-free.
git clone git@github.com:Mustry-Academy/cicd-lab-07-multi-gateway-deploy.git
cd cicd-lab-07-multi-gateway-deploy
git switch -c feature/wout-project
# add projects/wout/ with your welcome view …
git add projects/wout/
git commit -m "feat: add wout project with welcome view"
git push -u origin feature/wout-project
Observe

Only your project's checks run. Path-filtered CI: five people can open five PRs and nobody waits on anyone else's tests.

Observe

Merging deployed nothing to production. Main means "built and could ship", not "shipped". The only gateway that updated is your own test gateway, through the workflow you built in Part 1.

Part 2 · your project · B
2B
Tag your release and add your project to release.yaml to deploy it
Two moves: make the release exist, then point production at it. In that order, or the deploy has nothing to ship.
1
Tag it: git switch main → git pull → git tag <yourname>@v1.0.0 → push the tag. The tag is the release: a named version of your project that release.yaml can point at. Now the release exists.
2
Pin it: new branch, add one line to release.yaml: your project at v1.0.0. PR, approval, merge.
3
Watch the deploy action pick the job up on the site 7 runner and ship your project to production.
4
Verify live: open https://cloud.mustrysolutions.com, log in with admin / MergeIntoMain!, find your view.
# release.yaml
 gateway: oatmakers-site-7
 projects:
   oatmakers:        v2.0.0
+  wout:             v1.0.0
Observe

The deploy checks out your tag and copies your project's files to the gateway. Nothing gets built. And that is why the tag came first: release.yaml can only point at versions that exist.

Observe

The runner label routes the deploy: runs-on: [self-hosted, cicd-capstone]. Same repo, but only the runner carrying that label picks this job up. More sites would mean more labels, not more pipelines.

✅ Part 2 gate

Your project, your tag, your line in release.yaml, your view live on production. You just did the full loop every deploy in this lab will use.

The assignments
Tom lays the data foundation; Gregory builds the dashboard on top.
TOM
Set up the database connection, a migration with a seed, and a view on the data
  • Create the DB connection using the referenced secret POSTGRES_USERNAME. The secret is already on the environment: you reference it, you never see the value. Lab 06's secrets ladder, for real.

  • Build a Perspective view that queries the seeded data.
  • Verify on production: the connection is Valid, the migration ran, your view shows rows.
GREGORY
Historize the live tags into the database and build a dashboard
  • Enable tag history on Wout's tags, storing into Tom's database connection.
  • Build a dashboard view visualising that history.

Part 4 · extra · divide the work as a team
4 · EXTRA
Decide as a team who does what, then ship the extra challenges
Talk first: distribute the challenges below among the team. Everything from Part 3 is now shared infrastructure, and everything still ships the same way: small PR, quick review, next free tag, release.yaml.
CHALLENGE A
A showcase view for oatmakers
  • Build one nice-looking overview view that pulls the team's work together: the chart, the live tags and the database data on a single screen worth demoing.
  • Agree on the layout as a team, then land it section by section, in small PRs.
CHALLENGE B
A shared library, shipped at a version tag
  • Create a shared library project in projects/ with something the others can reuse: a styled component, a script function.
  • Tag it shared@v1.0.0 and pin it in release.yaml, so the oatmakers deploy ships with that specific version of your shared project. Use it from an oatmakers view to prove it.
CHALLENGE C
Build on the others, or get more reps
  • Extend what someone else shipped: more tags into the database, an extra migration, a chart on their data.
  • Or rebuild something that looks like their challenge, purely to train the move you want more practice on.
Be creative

These three are starters, not a script. Anything that practices what you learned today counts, as long as it ships by PR. Pull main often, keep PRs small, and solve the merge conflicts that show up: with five people in one project, they will.
