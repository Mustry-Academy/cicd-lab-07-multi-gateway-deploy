Part 1 · solo · ±20 min · exercises/lab.md
Choose the architecture and repository layout, and write it down

Every situation is different; this scenario has enough constraints to force real choices. Your deliverable is ARCHITECTURE.md: short, concrete, and defensible.

  • 1A: pick the target reference architecture and justify it in three sentences.
  • 1B: list the repositories: name, contents, who can see it. The vendor and the shared library must both have a home.
  • 1C: draw the file tree of your main repo and say where the runners live and which labels they carry.

Build the releases folder and run the v2.0.1 promotion as a PR

Not a sketch: real files, in your repo, changed through a real reviewed pull request. Your room-mate plays release approver.

  • 2A: build releases/ with dev/, test/ and prod/: one file per gateway, each pinning its projects at their own tags and the parent oatmakers-shared alongside, plus rollout.yaml.

Mustry Academy · CI/CD for IgnitionPart 2 · A → B
Part 2 · A
2A
Build the releases folder: one file per gateway, plus the rollout order
Full yaml examples are in exercises/lab.md. Everyone starts on v2.0.0: the hotfix comes in 2B.
1
On a branch, create the folders: releases/dev/, releases/test/, releases/prod/.
2
Prod: one yaml per gateway for at least sites 1 and 4 (backend + frontend each) and gw-central. Pin every project that runs there: frontend files carry all three, backend files the parent only. Each version is one of that project's own prefixed tags (oatmakers@v2.0.0 style), and the parent oatmakers-shared is pinned in every file, alongside. Copy the examples from the exercise file.
3
Test and dev: test/gw-test.yaml pinned like prod; dev/gw-dev.yaml is the odd one out: track: main, no pins.
4
rollout.yaml: three waves: canary (site 4, with a soak time), fleet, enterprise last.
5
PR, review against the checklist: every gateway has a file · the parent is pinned in each · the same oatmakers-shared version across all of prod (tested together = ships together) · dev is unpinned. Merge.
# releases/prod/gw-site4-frontend.yaml
gateway: gw-site4-frontend
projects:
  oatmakers:           v2.0.0   # its own tag
  oatmakers-packaging: v1.4.0
  oatmakers-shared:    v1.9.2   # the parent, alongside

# releases/dev/gw-dev.yaml
gateway: gw-dev
track: main      # every merge, automatically
The point

When this folder exists, "which version runs on site 4?" is a file you open, and "who changed it, when, why?" is git log. That's the whole trick.

Part 2 · B · the promotion
2B
Promote v2.0.1: canary PR first, then the fleet PR
The Oatmakers hotfix from day one, done the multi-gateway way. Your room-mate is the release approver; make them earn it.
1
The canary PR: on a branch, bump oatmakers to v2.0.1 in gw-site4-frontend.yaml only (the HMI runs on the frontend gateway). PR title: Promote oatmakers v2.0.1 to prod (wave 1: site 4).
2
The review: your room-mate checks four things: only site 4 changed · the pin matches the tagged hotfix (oatmakers@v2.0.1) · the parent version is untouched · the rollout order says site 4 goes first. Then they approve and you merge.
3
Narrate the machine: in the PR conversation, write the comment a pipeline would log: which runner picks this up, what it deploys (the project and its parent, to that one site), what it verifies during the soak, and which gateways stay unchanged.
4
The fleet PR: second branch, bump the remaining prod gateway files (site 1 and gw-central in your set). Review, merge.
5
Read your own audit trail: git log --oneline releases/prod/: two commits, two reviewed promotions, zero gateway logins.
# the canary PR diff, in full
releases/prod/gw-site4-frontend.yaml
-  oatmakers:           v2.0.0
+  oatmakers:           v2.0.1

# after both merges
git log --oneline releases/prod/
e41c2af Promote v2.0.1 (wave 2: fleet + central)
9b03d71 Promote v2.0.1 (wave 1: site 4)
✅ Part 2 gate

Two merged promotion PRs, each reviewed by your room-mate, and you can answer: how would you roll this back? (One word, starts with "revert".)

Solo · Part 3 · Tags
Part 3 · solo · ±15 min · exercises/lab.md
Write the tag-ownership policy for Oatmakers

Two incidents from the scenario land on your desk: the overwritten alarm limit, and the forty engineer-created tags that exist only on prod. TAGS.md is your answer.

  • Pick a master-of-tags model (development is master, production is master, or split ownership) and say in two sentences why it fits Oatmakers.
  • Write five concrete rules a pipeline could enforce: who may create tags where · what a deploy may overwrite · what it must never touch · how prod-created tags reach dev and the repo · what happens to operator setpoints on deploy day.
  • Name the cost. Every model trades something away: write one honest sentence about what yours gives up.
  • PR it; your room-mate reviews by attacking it with the two incidents: does each rule survive both?

Optional stretch
OPTIONAL
Stretch: four ways to go deeper
Pick by appetite. Each one extends a different part of the hour.
1
All eight sites, plus disaster recovery. Extend releases/prod/ to the full fleet and add gw-dr.yaml. Then the real question: which wave does DR belong in, and should it ever lag prod on purpose? Write your answer into rollout.yaml as a comment.
2
The vendor's release file. Add the palletizer project to the site 2 and site 7 release files, pinned to the vendor's own version numbers. Decide: may the vendor open promotion PRs in your releases repo, or do they hand versions over another way? Write the rule into ARCHITECTURE.md.
3
Cut the tags for real. In your design repo, create the prefixed tags your release files reference: git tag oatmakers@v2.0.0, oatmakers@v2.0.1, oatmakers-shared@v1.9.2. Then add one sentence to ARCHITECTURE.md: what does CI do with the prefix? (From the teaching: parse it, build only that project's folder, publish one artifact.)
4
Sketch the submodule. In ARCHITECTURE.md, sketch what changes if oatmakers-shared becomes a git submodule in a repo-per-site layout: which repos contain it, what the pinned commit means, and what a library upgrade looks like as PRs. No commands needed: it's the shape that matters.
