The assignment · ground rules
READ FIRST
How this works
  • Two repos, already set up for you. ignition-projects holds the three Ignition projects (oatmakers-shared is the inheritable parent). ignition-gateways holds one release.yaml per gateway: the desired state of the fleet.
  • The pipelines are already written. You will not write CI today. You will use it, and watch what it does. The "Observe" boxes on these slides are the actual exercise: stop and read what CI did before moving on.
  • Pre-flight: fork both repos with Actions enabled → cp .env.example .env → scripts/setup.sh (the four gateways and the runner come up) → scripts/validate.sh green.
  • You need your room-mate: the prod deploy requires a second approver, and that will be them.
  • The exercise file is the source of truth: exercises/lab.md has every command ready to copy. These slides are the map, not the territory.
  • Stuck? Re-read the matching teaching slide first, then say so in the room.
Ask early

Stuck for more than a couple of minutes? Say so in the room. Don't sit blocked in silence; that's exactly what we're here for.

The assignment · the mental model
MODEL
Two repos, two jobs: the seam between them is a version string
Keep this picture in your head for the whole hour. Every part of the exercise is one move across this seam.
ignition-projects
"What have we built?"
  • You merge, you tag, an artifact is published.
  • The tag prefix (oatmakers@v2.0.1) decides which project builds.
  • This repo does not know gateways exist.
ignition-gateways
"What is actually running, right now, where?"
  • One release.yaml per gateway: the desired state.
  • You change a version number in a YAML file, and a deploy happens.
  • Environments are directories on main, not branches.
The seam

The seam between the two repos is a version string. That's it. Nothing else crosses.

Source of truth

The gateway is not the source of truth. The YAML file is. If they disagree, the YAML file is right and the gateway is wrong, and the drift detector will say so.

The assignment · the plan
THE PLAN
One exercise in four parts: the same change travels the whole pipeline
Each part hands the next one its state. Do them in order; the timings are generous.
PART 1 · ±10 MIN

See the current state

Clone both repos. Read the fleet with matrix.py and the three independent version streams with git tag --list.

gate → you can name what runs where
PART 2 · ±15 MIN

Ship a change to one project

Fix an alarm label in oatmakers only. PR, merge, tag oatmakers@v2.0.1, read the artifact's manifest.

gate → one artifact, nothing deployed
PART 3 · ±20 MIN

Deploy it: dev → test → prod

Three one-line PRs in the gateways repo. Read the plan, merge to deploy, stage the prod rollout, watch drift appear in the matrix.

gate → v2.0.1 live on 3 gateways, gw-02 drifting
PART 4 · ±15 MIN

Change the shared parent

Add a function to oatmakers-shared. Watch every child get tested, deploy parent-first, then try to ship an untested pair and get refused.

gate → the manifest check blocks the bad deploy
Why this order

Part 1 shows you the aligned world you're about to disturb. Part 2 produces a version; Part 3 moves that version through the fleet; Part 4 changes the thing every project depends on, which is where the guardrails earn their keep.

Part 1 · see the current state
1
Read the fleet's state and the three version streams
Before you change anything, learn to read what is running where. Two commands, one grid, one tag list.
1
Clone both repos and run the matrix: cd ignition-gateways → python .ci/matrix.py. Everything is aligned. Enjoy it, it won't last.
2
List the tags: git tag --list in ignition-projects. Three independent version sequences in one repo: oatmakers is on v2.0.0 while oatmakers-packaging is on v1.4.0. They do not move together.
3
Prove the tag is per project: git show oatmakers@v2.0.0 --stat and git log --oneline oatmakers@v1.9.0..oatmakers@v2.0.0 -- projects/oatmakers/. The prefix is what tells CI which project to build.
$ python .ci/matrix.py
                 oatm.-shared  oatmakers  oatm.-packaging
gw-dev              v1.9.2     v2.0.0      v1.4.0
gw-test             v1.9.2     v2.0.0      v1.4.0
gw-oatmakers-01     v1.9.2     v2.0.0      v1.4.0
gw-oatmakers-02     v1.9.2     v2.0.0      v1.4.0

$ git tag --list
oatmakers@v1.9.0        oatmakers@v2.0.0
oatmakers-packaging@v1.3.0 … @v1.4.0
oatmakers-shared@v1.9.0 … @v1.9.2
✅ Part 1 gate

You can answer, from files alone: what runs where, and which three version streams exist. The gateway never had to be asked.

Part 2 · ship a change · A
2A
Change one project and watch CI build only that project
You fix something in oatmakers only. Nothing else should move, and you're going to watch CI agree.
1
Branch: git switch -c fix/oatmakers-alarm-label in ignition-projects.
2
Make a small, visible change in projects/oatmakers/: the alarm label your instructor points at. Commit, push, open the PR.
3
Watch the CI checks on the PR, then merge.
git switch -c fix/oatmakers-alarm-label
# edit projects/oatmakers/ …
git add projects/oatmakers/
git commit -m "fix: correct alarm label on line 2 overview"
git push -u origin fix/oatmakers-alarm-label
Observe

Only oatmakers tests run. oatmakers-shared and oatmakers-packaging are not built, not tested, not touched. That's path-filtered CI: poly-repo isolation, and you still keep everything in one place.

Observe

Merging deployed nothing. Main means "built and could ship", not "shipped". Nothing has moved yet.

Part 2 · ship a change · B
2B
Tag oatmakers@v2.0.1 and read the artifact's manifest
The tag is the release. Watch what CI does with the prefix, then open the file that comes back in Part 4.
1
Tag it: git switch main → git pull → git tag oatmakers@v2.0.1 → git push origin oatmakers@v2.0.1.
2
Watch the tag build: CI parses the prefix oatmakers, builds only that project, publishes oatmakers-v2.0.1.zip.
3
Read the manifest: python .ci/show-manifest.py oatmakers v2.0.1.
$ python .ci/show-manifest.py oatmakers v2.0.1
{
  "project": "oatmakers",
  "version": "v2.0.1",
  "commit":  "a3f9c21",
  "requires": { "oatmakers-shared": "v1.9.2" }
}
Observe

This is the important bit. The build recorded which oatmakers-shared it was compiled and tested against. Remember this: it comes back in Part 4.

✅ Part 2 gate

A merged PR, the tag oatmakers@v2.0.1, one published artifact with requires in its manifest, and still nothing deployed. You have an artifact. That's all.

Part 3 · deploy · A
3A
Deploy to dev: a one-line pull request in the gateways repo
Switch repos. This is where deploys happen, and the whole deploy is one line of YAML.
1
Branch in ignition-gateways: git switch -c deploy/oatmakers-v2.0.1-dev.
2
Edit gateways/dev/gw-dev/release.yaml: bump oatmakers from v2.0.0 to v2.0.1. One line. Commit, push, open the PR.
3
Read the plan on the PR, then merge and watch the deploy: pull oatmakers-v2.0.1.zip, ship to gw-dev, smoke test.
4
Verify: open the dev gateway and find your alarm label, live.
# gateways/dev/gw-dev/release.yaml
 gateway: gw-dev
 projects:
   oatmakers-shared:    v1.9.2
-  oatmakers:           v2.0.0
+  oatmakers:           v2.0.1
   oatmakers-packaging: v1.4.0
Observe

The PR check runs a plan. It tells you exactly what will change on which gateway before you merge. Read it. Then merge, and now it deploys.

The pattern

The PR is the deploy request. The approval is the change control. The merge is the trigger. You will use this exact move for every deploy from here on.

Part 3 · deploy · B
3B
Promote to test, then one prod gateway: drift becomes visible
Promotion is the same version number moving right through a sequence of files. Not a merge between branches.
1
Test: same one-line bump in gateways/test/gw-test/release.yaml, new branch, PR, merge.
2
Prod, one gateway only: bump gateways/prod/gw-oatmakers-01/release.yaml. The prod PR requires a second approver: get your room-mate. Note the rule is path-based: it applies to prod/ and not to dev/.
3
Merge and watch it deploy to gw-oatmakers-01 and only that one. gw-oatmakers-02 is untouched: staged rollout, least critical first, bake, then the rest.
4
Run the matrix again and find the drift arrow.
$ python .ci/matrix.py
                 oatm.-shared  oatmakers  oatm.-packaging
gw-dev              v1.9.2     v2.0.1      v1.4.0
gw-test             v1.9.2     v2.0.1      v1.4.0
gw-oatmakers-01     v1.9.2     v2.0.1      v1.4.0
gw-oatmakers-02     v1.9.2     v2.0.0 ←    v1.4.0
Notice

Environments are directories, all on main. If you'd made them branches, you'd now be cherry-picking and fighting merge conflicts between dev and prod. You aren't.

✅ Part 3 gate

v2.0.1 live on dev, test and gw-oatmakers-01 through three one-line PRs, the prod PR carries a second approval, and the matrix shows gw-oatmakers-02 drifting. Drift being visible is the point of the matrix.

Part 4 · the parent · A
4A
Change the parent project and watch every child get tested
This is the interesting part. A change to oatmakers-shared is a change to every project that inherits from it, and CI knows.
1
Branch in ignition-projects: add a function to oatmakers-shared's script library (format_shift_timestamp, per the exercise file). PR.
2
Read the PR carefully, then merge and tag: git tag oatmakers-shared@v1.10.0, push the tag.
3
Deploy to dev: bump oatmakers-shared to v1.10.0 in gw-dev/release.yaml. PR, plan, merge.
Observe

All three test suites run this time. A change to the parent is a change to every child, and CI can see the whole graph because they live in one repo.

Observe

The impact report posts on the PR. It statically scans every project for call sites of what you touched. This answers "I cannot possibly know what I just broke."

Observe

The parent deploys first. The pipeline topologically sorts: if it shipped oatmakers first, it would briefly reference a library that isn't there. And note what you did NOT do: you never bumped the children. The library moved; they didn't.

And it passed

This deploy passed the manifest check even though oatmakers@v2.0.1 was built against v1.9.2: a compatible minor bump (v1.9.2 → v1.10.0) is allowed. Semver doing its job. Now let's find the edge.

Part 4 · the parent · B
4B
Try to ship an untested combination and watch the manifest check refuse
Now break it, on purpose. Pair a major parent bump with a child that has never seen it.
1
Get the breaking tag from your instructor: oatmakers-shared@v2.0.0, a major bump with a renamed function.
2
Try to deploy it to prod: bump oatmakers-shared to v2.0.0 in gw-oatmakers-01/release.yaml, leaving oatmakers at v2.0.1, which was built against v1.9.2. Open the PR.
3
Read the refusal, line by line. Then close the PR without merging: that red cross just saved your 02:00.
✗ gw-oatmakers-01
  oatmakers@v2.0.1 was built against
    oatmakers-shared@v1.9.2
  gateway declares
    oatmakers-shared@v2.0.0
  Major version mismatch.
  This combination has never been tested.
✅ Part 4 gate · the payoff

The artifact remembered what it was built against, and the pipeline refused to ship a combination nobody has ever run. Without the manifest, that deploy succeeds and breaks at 2am. This is what Part 2's requires line was for.

Optional stretch
OPTIONAL
Stretch: three ways to go deeper
Pick by appetite. All three stay inside the two repos you already have.
1
Finish the rollout. Promote v2.0.1 to gw-oatmakers-02, and read rollout.yaml: write one sentence in the PR on why gw-01 went first. The matrix goes fully green again.
2
Roll back. git revert the gw-oatmakers-01 deploy PR and watch the pipeline put v2.0.0 back. Nobody logs into a gateway: the file is the state, in both directions.
3
Read the plan like an operator. Open any merged deploy PR and reconstruct, from its plan and logs alone: what changed, where, when, and who approved it. That's the audit trail: no archaeology.
# stretch 2: rollback is a revert
git revert <merge-commit-of-the-deploy-PR>
git push
# the pipeline reads the file and makes it
# true again:
gw-oatmakers-01  oatmakers  v2.0.1 → v2.0.0
Same machinery

All three stretches are the same one-line-PR move you've made all hour. That's the point: once state is files, every operation is the same operation.
