Warm-up · together
WARM-UP · 1
Secret triage
The three-boxes slide, applied to your own clone: what here is public, per-environment, secret, and is the secret stuff actually ignored?
1
Hunt for candidates. Grep the repo for secret-shaped values (see command). Every hit goes in one of the three boxes.
2
Sort them in NOTES.local.md: public / per-environment / secret. Defaults like ${POSTGRES_PASSWORD:-ignition}: which box, and why?
3
Check the fence: git check-ignore must confirm .env and secrets/ contents are ignored, and the *.example files are not.
4
Run the scanner: scripts/validate.sh includes a secret scan (gitleaks). Zero findings = the baseline you must protect all lab.
# 1: secret-shaped strings in tracked files
grep -rniE "password|api_key|token|secret" \
  --exclude-dir=.git .

# 3: is the fence up?
git check-ignore -v .env secrets/postgres_password.txt
git check-ignore secrets/postgres_password.txt.example \
  && echo "BAD: example must be tracked!"

# 4: the scanner
scripts/validate.sh        # secret scan → 0 findings
Three boxes

Public → committed plain. Per-environment → committed default + env var. Secret → never committed. Sorting is 80% of secrets management.

WARM-UP · 2
Leak & rotate drill
The We-do demo, now in your hands: leak a fake key on a scratch branch and prove to yourself that Git never forgets.
1
Scratch branch: git switch -c drill/leak. Commit an obviously fake key (see snippet).
2
"Fix" it the way everyone does: delete the file, commit again. Feels solved.
3
Prove it isn't: git log -p reads the key straight back out of history. It would be in every clone and fork, forever.
4
Write the incident response in NOTES.local.md, two lines: 1. rotate the credential, 2. then (best-effort) scrub history. In that order, always.
5
Clean up: back to main, delete the scratch branch. Don't push it.
git switch -c drill/leak
echo 'API_KEY="ak_live_0badc0ffee"' > oops.txt
git add oops.txt && git commit -m "wire up historian"

git rm oops.txt && git commit -m "remove key"  # "phew"

git log -p -- oops.txt | grep ak_live
+API_KEY="ak_live_0badc0ffee"   ← still there

git switch main && git branch -D drill/leak
Iron rule

A pushed secret is a burned secret. Deleting the line (even force-pushing) without rotating is theatre.

Solo · Phase 1
Phase 1 · solo · climb the secrets ladder
One credential, two rungs up

You'll take the Postgres password (rung 1 since Lab 02) and move it up the ladder on your own stack, exactly like the We-do demo.

  • Part A: from Compose environment: to a Docker secret at /run/secrets/.
  • Part B: an Ignition secret provider, and the gateway DB connection re-pointed at a referenced secret.
  • Gate: scripts/validate.sh green, secret scan finding zero plaintext secrets in tracked files.
Mustry Academy · CI/CD for IgnitionPhase 1 · Part A → B → gate
Phase 1 · Part A
PART A
Postgres password → Docker secret
Rung 1 → rung 2: the value leaves the container's environment and becomes a mounted file.
1
See the leak first: docker inspect the database container and find POSTGRES_PASSWORD in plaintext. That's what you're fixing.
2
Create the secret file from its committed example: secrets/postgres_password.txt (gitignored).
3
Edit docker-compose.yaml: add the top-level secrets: block, attach it to the DB service, and swap POSTGRES_PASSWORD for POSTGRES_PASSWORD_FILE.
4
Recreate the service and prove the leak is gone: inspect shows no password, but the file is there at /run/secrets/postgres_password.
# docker-compose.yaml: the change
secrets:
  postgres_password:
    file: ./secrets/postgres_password.txt

services:
  timescaledb:                # the DB service
    secrets: [postgres_password]
    environment:
      POSTGRES_PASSWORD_FILE: /run/secrets/postgres_password

# prove it (steps 1 & 4)
docker compose up -d --force-recreate timescaledb
docker inspect lab07-timescaledb | grep -i postgres_
docker compose exec timescaledb \
  cat /run/secrets/postgres_password
Careful

Use the same password value as before. Postgres only sets the password on first volume init, so changing the value here won't change the running database's password.

Phase 1 · Part B · the gate
PART B
A referenced secret in the gateway
Rung 3: gateway config stops containing the credential and starts referencing it. This is what makes your committed config portable.
5
Create a secret provider on your local gateway's web UI (Config → Secrets) that reads the file /run/secrets/postgres_password.
6
Re-point the DB connection: edit the gateway's database connection so the password is the referenced secret, not a pasted value. Test the connection: still Valid.
7
Prove nothing leaked: export the gateway config as in Lab 04's file-based flow, then grep the export for the password. You should find a reference by name: no plaintext, no ciphertext.
8
The gate: scripts/validate.sh green, with the secret scan at zero findings. Note in NOTES.local.md: why would this exact config now work on dev and prod unchanged?
# 7: the export must not contain the value
grep -rn "$(cat secrets/postgres_password.txt)" \
  ignition-data/ # or your exported config dir
# no matches = nothing leaked

grep -rn "postgres_password" ignition-data/config.json
# → a reference by NAME is fine; that's the point

# 8: the gate
scripts/validate.sh        # → ✅  secret scan: 0
✅ Gate

Embedded = pet gateway. Referenced = pipeline. Green here means your config can travel local → dev → prod, and you're cleared for Phase 2.

Phase 2 · solo · deploy the database with your code
A schema change rides the pipeline

Remember: db-init/ is bootstrap. It will never touch an existing database. Change ships as a migration, and the pipeline applies it before your screens arrive.

  • Part C: write V2__add_downtime_log.sql, apply it locally, read the history table.
  • Part D: the migrate step in deploy.yml, then a PR to develop and a green run: migrate → ship → scan → verify.
Mustry Academy · CI/CD for IgnitionPhase 2 · Part C → D
Phase 2 · Part C
PART C
Write V2__…, migrate locally
Your first versioned migration: a new table the example project's screen will read.
1
Write the file: migrations/V2__add_downtime_log.sql. Versioned prefix, double underscore, description: the name is a contract.
2
Apply it: scripts/migrate.sh against your local database.
3
Read the ledger: query flyway_schema_history to find V1 (baseline) and your V2, each with a checksum. Run migrate.sh again: nothing reapplies.
4
See it in the gateway: open the example project's downtime screen, which reads the new table. Empty is fine; erroring is not.
5
Feel the immutability rule: edit your applied V2 (add a space), re-run migrate.sh → checksum mismatch, hard fail. Undo the edit. Changes go in V3, never in V2.
-- migrations/V2__add_downtime_log.sql
CREATE TABLE downtime_log (
  id      BIGSERIAL PRIMARY KEY,
  ts      TIMESTAMPTZ NOT NULL DEFAULT now(),
  reason  TEXT NOT NULL,
  minutes INT
);

# apply + inspect the ledger
scripts/migrate.sh
docker compose exec timescaledb psql -U ignition \
  -d ignition_dev -c "SELECT version, description
  FROM flyway_schema_history;"
git log for schema

Versioned · append-only · checksummed. V1 is never edited; history only grows. Same instinct as commits.

Phase 2 · Part D
PART D
Watch the pipeline migrate dev before it ships
The whole point of the afternoon: schema and screens travel in the same PR, and the database is ready before the gateway scans.
6
Open .github/workflows/deploy.yml and add/enable the migrate step, before the ship-files step, targeting the dev database.
7
Branch, commit, PR: migration + workflow change on a feature branch, PR to develop. CI must be green (your secret scan runs here too).
8
Merge and watch the run on the self-hosted runner: validate → migrate dev DB → ship files → scan → verify. Find the migrate step's log lines.
9
Prove it: query flyway_schema_history in the dev database: V2 applied by the pipeline, not by you. Check the dev gateway's screen.
git switch -c feature/downtime-log
git add migrations/V2__add_downtime_log.sql \
        .github/workflows/deploy.yml
git commit -m "feat(db): downtime_log + migrate step"
git push -u origin feature/downtime-log
gh pr create --base develop --fill

# after merge: watch it go
gh run watch
# migrate dev DB → ship → scan → verify ✅
Order is the point

Old screens tolerate a new table; new screens don't tolerate a missing one. If migrate fails, the run stops: no screens ship. That's the safety property you just built.

Optional stretch
OPTIONAL · S1
Stretch 1: rename a column, break nothing
The higher-value one. A rename is a drop in disguise, so do it as two migrations with a working system in between.
1
Expand: in V3__downtime_add_reason_code.sql, add the new column and copy the data over. Old screens still read the old column: nothing breaks.
2
Switch the readers: update the screen/named query to the new column, ship it through the pipeline (same PR as V3, or after).
3
Contract: only when nothing reads the old column, V4__downtime_drop_reason.sql drops it. In real life this ships releases later.
4
Ask yourself: at which points between V3 and V4 could a gateway run old screens against the new schema, and does anything break? (It shouldn't. That's the choreography.)
-- V3 (expand): add + backfill, drop nothing
ALTER TABLE downtime_log ADD COLUMN reason_code TEXT;
UPDATE downtime_log SET reason_code = reason;

-- …screens switch to reason_code, deploy, soak…

-- V4 (contract): only once nothing reads it
ALTER TABLE downtime_log DROP COLUMN reason;
Never

Never rename in one step while gateways are live: between the migration and the screen deploy, every query against the old name 500s.

OPTIONAL · S2 + S3
Seed data as code · scanner in CI
Two smaller ones. Pick whichever gap bothers you more.
S2
Repeatable seed migration. Put reference data in migrations/R__seed_reference_data.sql: idempotent SQL (upserts). Run migrate.sh twice, then edit the file and run again. When do repeatables re-run? Check the history table for the answer.
S3
Secret scan in CI. Add a secret-scan job to .github/workflows/ci.yml so a leaked key fails the PR. Test it: PR a fake key from Warm-up 2 → red ❌ → close the PR without merging.
-- S2 · R__seed_reference_data.sql (idempotent!)
INSERT INTO downtime_reasons (code, label)
VALUES ('MECH','Mechanical'), ('ELEC','Electrical')
ON CONFLICT (code) DO UPDATE SET label = EXCLUDED.label;

# S3 · ci.yml: new job
secret-scan:
  runs-on: ubuntu-latest
  steps:
    - uses: actions/checkout@v4
      with: { fetch-depth: 0 }   # full history!
    - uses: gitleaks/gitleaks-action@v2
Why fetch-depth 0

The scanner must see history, not just the tip. Warm-up 2 showed you exactly why a tip-only scan misses the burned key.
