Deployments in a multi-gateway architecture
Everything so far deployed one repository to one gateway. Real plants have backend and frontend gateways, eight sites, an enterprise level, vendors, and one question nobody can answer from memory: which version runs where? This session lays out the architectures, the repository layouts that map onto them, the releases folder that does the bookkeeping, and the tag-ownership decision you must make before any of it goes live.

Teaching
01
Every situation is different: this session gives you options, not one answer
Gateway count, site count, team boundaries, vendors, IT constraints: no two plants match. The goal today is that you can pick what is right for your situation, and defend the pick.
  • You get the menu, not a prescription. Three reference architectures, the repository layouts that map onto each, a deploy mechanism that scales past one gateway, and the tag-ownership question. We will not go deep into any single one; we make sure you know they exist and what each is for.
  • You already own every building block. Labs 03 to 06 gave you pipelines, self-hosted runners, file-based and image-based deploys, secrets, and db-migrations. Nothing today needs a new tool.
  • What changes with more gateways is the routing, not the deploy. The new questions are: which files go to which gateway, triggered by what, in which order, and who is allowed to decide.
Where the course left us

One repository, one site, one logical gateway in three environments (local, dev, prod). Keep that picture in your head: every slide today is a controlled change to it.

Part 1 · Architecture 1
02
Standalone: one gateway does everything
Devices, tags, history, Perspective sessions: all on one gateway. Most plants start here, and at the right size it is a perfectly good place to stay.
  • One box, one failure domain. PLC polling, history and every client session share the same CPU and the same restart.
  • Where it strains: heavy Perspective load competes with device polling, and every deploy touches the box that also runs production.
  • The CI/CD view: one repo, one pipeline, one deploy target. Labs 04 to 06 already built everything this architecture needs.

Part 1 · Architecture 2
03
Frontend and backend gateways: split the humans from the plant floor
Backend gateways own device connections, tag providers and history. A frontend gateway owns the Perspective sessions. They talk over the gateway network.
  • Why split: client sessions can no longer starve device polling, and you can redeploy or restart visualization without touching data collection.
  • It scales: more frontends behind a load balancer, more backends per device group, without redesigning anything.
  • The CI/CD view: one plant, but several deploy targets with different content: projects mostly go to the frontend, device and tag config to the backends. The pipeline has to route. That routing is the new skill of this lab.

Part 1 · Architecture 3
04
Multi-site with an enterprise level: hub and spoke
Each site collects and runs locally; a central level aggregates history, reporting and corporate clients. This is where Oatmakers with its eight sites is heading.
  • Sites keep local autonomy. Edge or standard gateways collect and buffer at the plant; local clients keep working when the WAN link drops.
  • The center aggregates. A central database service, enterprise reporting, and corporate clients through load-balanced frontend gateways.
  • The CI/CD view: many targets in different networks. A self-hosted runner per site becomes the deploy agent inside each network, and "which version runs where" becomes a real bookkeeping problem. Both get solved this hour.

Part 3 · The releases folder
07
A releases folder records which version runs on which gateway
Usually its own small repository. One folder per environment, one file per gateway, and every file is plain text under version control: the fleet's state has a git log.
📦 oatmakers-releases (a separate repo)

A promotion is a pull request in the releases repo
The Oatmakers hotfix from day one, done fleet-wide: bump the pinned version, let review be the approval gate, and let rollout.yaml order the waves.

(the HMI runs on the frontend gateway)

+  oatmakers:  v2.0.1
Review = the approval gate

The reviewer approves a promotion, not code. Merge is the trigger; nobody logs into a gateway.

Rollback is git revert

Revert the release PR and the pipeline makes the old pins true again: the file is the state.

Part 3 · Multi-site · option B
10
Multi-site option B: splitting into multiple repositories
Four splits you can make, each answering a different question: team boundaries, vendor access, infra ownership, shared code. Mix them freely.

A repository per site, plus one for the enterprise level

oatmakers-site1 … oatmakers-site8 · oatmakers-enterprise

Each site team owns its own repo and its own pipeline to its own gateways; the enterprise repo deploys gw-central. Cost: a change every site needs becomes eight PRs, and sites can drift apart.

oatmakers-site4
site 4 gateways only

A repository per vendor project

palletizer-vendor

One project, nothing else. The vendor works in it and their pipeline deploys only that project to your gateways, alongside your own deploys. They never see the rest of the plant. Project-level deploys (lab 04) are what make this safe.

palletizer project
gw-site2 · gw-site7

Config repositories separate from one projects repository

oatmakers-projects · oatmakers-config-site1 …

All projects combined in one repo keeps the shared library and cross-project review easy; gateway config per site lives in repos the infra team owns. Cost: a feature that needs both takes two PRs.

oatmakers-shared · pinned per repo

Part 3 · Choosing
11
Choosing a repository layout: the trade-offs side by side
Pick the fewest repositories that answer your actual questions: who may see what, who owns what, and how shared code travels.
LayoutStrongest whenWatch out for
One repository for everything One team; the shared library is used everywhere; a cross-gateway change should be one reviewable PR. Everyone with access sees everything; vendors need carve-outs; the repo and its churn grow with the fleet.
Repository per site + enterprise Site teams work at their own pace; access ends at the site boundary. A shared change is N pull requests; without bookkeeping the sites drift apart silently.
Vendor repository per project External parties must contribute without seeing the plant. Needs disciplined project-level deploys and a clear interface contract (which tags and queries the project may expect).
Config repos + one projects repo Infra and application are different teams; all projects stay together so the shared library stays easy. One feature can need two PRs in two repos, in the right order.
Whatever you pick

More repos and more gateways all raise the same question: which version runs where? The releases folder from the start of this part answers it, for every layout on this table.

Part 4 · The counter-current
Tag management: who is the master of tags?
Tags are the one data stream that also flows against the deploy direction: they change on production, while everything else changes in Git. Decide ownership before your first multi-gateway deploy, not after the first overwritten setpoint.
Mustry Academy · CI/CD for Ignitiontwo scenarios · three ownership models
Part 4 · Tag ownership
12
Decide who owns the tags: the development team or production
Two things that really happen on plant floors, and the three ownership models that deal with them. Your pipeline must enforce whichever you pick.
Scenario 1 · operators change tags on production

Night shift trims an alarm limit at 03:00 to stop a nuisance alarm. Saturday's deploy ships the tag export from Git: the limit is silently back to the old value. If operators may change tags on production, your deploy must not blindly overwrite them.

Scenario 2 · process engineers create tags on production

Forty new tags, born directly on the prod gateway. Dev has never seen them: screens can't be tested against them, and a strict "make it match Git" deploy would even delete them. Tags born on production need a way back into dev and the repo.

Model 1

Development is master

Every tag change travels through the repo and the pipeline. Editing tags on production is forbidden (or exported back the same day). The deploy may overwrite the tag provider.

Fits when → tags change rarely and the dev team owns the process
Model 2

Production is master

Tags are exported from prod into the repo on a schedule; the deploy never touches the tag provider. Dev refreshes itself from those exports, so scenario 2's tags reach dev too.

Fits when → engineers and operators live in the tag browser
Model 3

Split ownership

Dev owns structure (UDT definitions, folders, new tags); production owns values (setpoints, alarm limits). The deploy ships definitions and never writes values: or you split by provider or folder.

Fits most plants → but only if the split is written down and enforced
The rule

There is no universal answer, but there must be an answer: written down, agreed with operations, and enforced by the pipeline (what a deploy may overwrite, what it must merge, what it must never touch).
