Part 1 · Secrets
02
Sort every value into one of three boxes
Sorting is 80% of secrets management. Every value in your project is public, per-environment, or secret — and each box has a different home.
Kind 1

Public

Port numbers, module lists, project names, timeouts, modules.json.

→ committed, plain, in the repo
Kind 2

Per-environment

Gateway URLs, database names, container names. Same shape, different value per env.

→ committed defaults + env vars / GitHub environment variables
Kind 3

Secret

Passwords, API keys, PATs, private keys, keystores. Anyone holding the value is you.

→ never committed — .env, Docker secrets, secret providers, GitHub secrets
The classic failure

A secret masquerading as per-environment config — pasted into docker-compose.yaml or a gateway export "just to make it work", then committed, then pushed. Once it's pushed, it's a different problem entirely →

Part 1 · Secrets · the iron rule
03
A pushed secret is a burned secret
Lab 01 taught you commits are immutable snapshots. Here's the dark side of that superpower.
# Monday, 09:14 — "just to make it work"
git commit -m "wire up historian"
+ IGNITION_API_KEY = "ak_live_9f3ce81b..."
# Monday, 09:40 — "phew, removed it"
git commit -m "oops, remove key"
- IGNITION_API_KEY = "ak_live_9f3ce81b..."
# anyone, any clone, any time later:
git log -p | grep ak_live
+ IGNITION_API_KEY = "ak_live_9f3ce81b..."  ← still there
The only correct response

1. Rotate the credential — immediately. The old value is public property now; make it worthless.

2. Then, optionally, scrub history (git filter-repo, BFG). On a shared remote this is best-effort: forks, clones and CI caches keep copies.

Deleting the line and force-pushing without rotating is theatre.

Prevention beats response

Secret scanners (e.g. gitleaks) in scripts/validate.sh and CI catch the Monday-09:14 commit before it's pushed. You'll wire this in the assignment.

Step through the leak everyone commits once — and why "removing" it fixes nothing.
Part 1 · Secrets · the ladder
04
The secrets ladder
Five rungs, bottom to top. Each rung fixes the leak of the one below it. You don't need the top rung everywhere — you need to know which rung you're on.
0
Hardcoded in the repo plaintext in docker-compose.yaml or committed config
Leaks into: Git history, every clone, every fork. Never acceptable.
1
Env vars via .env + Compose interpolation what labs 02–06 already do
Out of Git. Still leaks into: docker inspect, compose config, the container's process env.
2
Docker secrets files mounted read-only at /run/secrets/<name>
Not in inspect, not in process env. Still a plaintext file on the host you must protect.
3
Ignition 8.3 secret providers gateway config references a secret instead of containing it
Config becomes portable across local / dev / prod — the CI/CD answer. Next slides.
4
External secret managers Vault, AWS/Azure secret stores — central rotation & audit
Same referenced-secret idea, org-scale. Know it exists; out of scope today.
Climb the ladder rung by rung — bottom to top.
Part 1 · Secrets · rung 1
05
Rung 1 — .env files, done properly
You've been on this rung since lab 02 without ceremony. Let's make the convention explicit — it's good, and it has a known ceiling.
  • .env is gitignored; .env.example is committed. The example documents every variable with a safe placeholder. Every lab repo ships one.
  • Compose interpolates ${VAR:-default} at parse time — safe defaults for the lab, real values from your .env.
  • The ceiling: the value still lands in the container's environment — visible in docker inspect, docker compose config, and to any code that can dump env.
# docker-compose.yaml — lab 04/05, unchanged
environment:
  GATEWAY_ADMIN_PASSWORD: "${GATEWAY_ADMIN_PASSWORD_LOCAL:-lab04password}"
  POSTGRES_PASSWORD: "${POSTGRES_PASSWORD:-ignition}"

# where the leak shows:
docker inspect lab07-timescaledb \
  | grep POSTGRES_PASSWORD        # plaintext
Verdict

Perfectly fine for a laptop lab. Not where you stop for shared dev/prod machines — that's rung 2.

Part 1 · Secrets · rung 2
06
Rung 2 — Docker secrets: a file, not an env var
A top-level secrets: block turns a host file into a read-only file at /run/secrets/<name> inside the container. No Swarm required for file-backed secrets.
# docker-compose.yaml
secrets:
  postgres_password:
    file: ./secrets/postgres_password.txt   # gitignored

services:
  timescaledb:
    secrets: [postgres_password]
    environment:
      # official postgres image: _FILE suffix
      POSTGRES_PASSWORD_FILE: /run/secrets/postgres_password
  • Not in docker inspect, not in the process environment — the value only exists as a mounted file the process reads at startup.
  • The _FILE convention: many official images (Postgres included) accept VAR_FILE pointing at a file instead of VAR.
  • Repo convention: secrets/ is gitignored; a committed *.example file documents each secret — same idea as .env.example.
  • Still: a plaintext file on the host. And Ignition config itself (DB connections, auth) needs its own answer — that's the next rung.

Part 1 · Ignition 8.3 · secret providers
07
Embedded vs referenced secrets
Gateway config (DB connections, auth profiles, email…) needs credentials. 8.3 gives you two ways to store them — the difference decides whether your config can travel.
Embedded secret
  • The value is encrypted by this gateway and stored inside its own config.
  • Fine for a single, hand-managed gateway — set it once in the UI, done.
  • The catch: the ciphertext is bound to that gateway's encryption key. An exported config.json carrying it is not portable to another gateway.
  • For a pipeline that promotes the same config local → dev → prod: a dead end.
Referenced secret
  • Config stores a reference: "get db_password from provider X" — never the value.
  • A secret provider resolves it at runtime — from an environment variable or a file.
  • A file provider + Docker secrets = rungs 2 and 3 snap together: the provider reads /run/secrets/db_password.
  • Exported config contains no secret material — it can be committed, diffed, reviewed and deployed.
Rule of thumb

Embedded = pet gateway. Referenced = pipeline. If a gateway's config is in Git (and after labs 04–06, yours is), its secrets must be referenced.

Part 1 · Ignition 8.3 · portability
08
One config, three gateways, three values
The same committed config works everywhere, because each environment injects its own value under the same secret name. Config stays in Git; values stay out.
📄 config.json — committed, in the repo
"database-connections": {
  "plant_db": {
    "username": "ignition",
    "password": { "ref": "db_password" }  ← a name,
                                      not a value
  }
}
→
local resolves db_password → /run/secrets/db_password = your laptop's value
dev resolves db_password → the dev stack's value
prod resolves db_password → the prod value — which only prod ever holds
Same pattern, everywhere

You already use this model in CI: one workflow file, and the lab-gateway-dev / lab-gateway-prod GitHub environments each supply their own IGNITION_API_KEY. Referenced secrets bring the identical idea inside the gateway.

Part 1 · Secrets · in the pipeline
09
Secrets in GitHub Actions — the rules you already half-know
Lab 04 made you create environment secrets. Here's the discipline that goes with them.
  • Environment-scoped, not repo-wide. lab-gateway-prod holds the prod key; a PR build against dev can't even see it. Add required reviewers on the prod environment for a human gate.
  • Masking is a seatbelt, not a licence. Actions replaces known secret values with *** in logs — but don't echo them, don't set -x around them, don't write them into artifacts.
  • Least scope, short life. Per-gateway API keys, not one master key. PATs with repo scope only. Rotate on any doubt — rotation should be boring.
  • Forks don't get secrets. pull_request runs from forks see empty secrets by design — one more reason deploys trigger on push/tag, never on PR.
# deploy.yml — lab 04, revisited
jobs:
  deploy:
    runs-on: [self-hosted, lab07]
    environment: lab-gateway-dev   # ← scoping
    steps:
      - name: Trigger scan
        env:
          API_KEY: ${{ secrets.IGNITION_API_KEY }}
        run: scripts/trigger-scan.sh --gateway dev
Never

run: echo ${{ secrets.IGNITION_API_KEY }} — and never interpolate a secret straight into a shell command line; pass it through env: so it stays out of the workflow's parsed command echo.

Part 1 · Secrets · the list
10
The never-commit list for an Ignition repo
Print this. Tape it to the monitor of whoever owns the repo.
  • .env — any real one. (.env.example with placeholders: yes, please.)
  • secrets/ contents — only the *.example files are tracked.
  • PATs & API keys — ghp_…, runner registration tokens, gateway API keys.
  • Gateway backups (.gwbk) from real systems — they carry users, DB connections and keystores in one convenient bundle.
  • Keystores / TLS private keys — ssl.pfx, metro-keystore, *.p12.
  • Exports with embedded-secret ciphertext — encrypted, but non-portable and secret-shaped. Use referenced secrets so this never comes up.
Enforcement, not vigilance

Humans forget; hooks and CWe-don't. This repo's scripts/validate.sh runs a secret scan (gitleaks) and ci.yml repeats it on every PR — a leaked key fails the build before a reviewer ever sees it.

Schema drift — the failure mode
The deploy pipeline from lab 04 ships the screen. Nothing ships the table the screen needs. That gap has a schedule: it bites on release day.
📄 what Git deploys to prod
Perspective view "DowntimeLog"
  ↳ named query: SELECT reason, minutes
    FROM downtime_log ...   ← new screen,
                              shipped ✅
🗄 what the prod database has
ignition_prd=# \dt
  batch_log      ✓
  recipe_params  ✓
  downtime_log   ✗ — created by hand
                   on dev only, three
                   weeks ago
  • "Worked on dev." Because someone manually created the table on dev. Prod was never told. The screen 500s on release night.
  • No history. Who changed the column type? When? Why? With hand-run SQL there is no equivalent of git log for your schema.
  • The fix is the same fix as always: make schema change files in the repo, reviewed in the same PR as the screens that need them, applied by the pipeline.

Part 2 · Database · what you already have
12
db-init/ — bootstrap, not deployment
Labs 04/05 already mount SQL into the database container. Time to be precise about what that mechanism can and cannot do.
# docker-compose.yaml — labs 04/05, unchanged
volumes:
  - ./db-init:/docker-entrypoint-initdb.d:ro

# db-init/01-create-env-databases.sql
CREATE DATABASE ignition_dev;
CREATE DATABASE ignition_prd;
The one rule

Postgres runs /docker-entrypoint-initdb.d once — on first init of an empty data volume. Volume already exists? Nothing runs. Ever again.

  • Great for: creating the databases, the app user, extensions — the "day zero" shape of a fresh environment.
  • Useless for: changing a database that already has data. Editing db-init/*.sql after first boot changes nothing on any running environment.
  • The trap: "I added the table to db-init and it worked!" — because you also wiped your volume. Prod doesn't get its volume wiped.
  • So: day-zero shape → db-init/. Day-one-and-beyond change → migrations →

Part 2 · Database · migrations
13
Migrations — git log for your schema
Numbered, immutable SQL files in the repo. A runner applies the ones a database hasn't seen yet, in order, and writes down what it did.
📄 migrations/V1__baseline.sql
CREATE TABLE batch_log (…);
CREATE TABLE recipe_params (…);
📄 migrations/V2__add_downtime_log.sql
CREATE TABLE downtime_log (
  id BIGSERIAL PRIMARY KEY,
  reason TEXT, minutes INT, …);
📄 migrations/V3__downtime_add_area.sql
ALTER TABLE downtime_log
  ADD COLUMN area TEXT;
flyway migrate
→
applies what's
missing, in order
flyway_schema_history — inside each database
V1baseline · applied 2026-07-01 · checksum 8a41…
V2add downtime log · applied 2026-07-08 · checksum c93f…
V3downtime add area · pending on prod — applies on next deploy
Sound familiar?

Versioned, append-only, hash-checked history that different environments sync from at their own pace. It's the Git mental model, applied to your database — the history table is the reflog, migrate is the pull.

Part 2 · Database · the rules
14
Four rules that keep migrations boring
Boring is the goal. Every rule exists because someone's release night didn't have it.
  • 1 · Applied migrations are immutable. The runner stores a checksum; editing V2__… after it ran anywhere makes every other environment fail loudly. Need a change? Write V4. (Commits are immutable too — same instinct.)
  • 2 · Roll forward, not back. A "down" script can't un-drop data. The recovery for a bad migration is a new migration that fixes it — plus the backup you were taking anyway.
  • 3 · Expand → migrate → contract. Zero-downtime change is choreography: add the new column (old screens still fine) → deploy screens using it → remove the old column releases later. Never rename in one step while gateways are live.
  • 4 · Schema and screens travel in the same PR. The migration that creates downtime_log reviews alongside the view that queries it. One PR, one story, no drift.
Seed data

Reference data the app needs (enum tables, default recipes) ships as migrations too — versioned like everything else. Test/demo data does not: that's per-environment tooling, never a prod migration.

Smoke check: gateway up, query the new table.

Why migrate before ship

Old screens tolerate a new table; new screens don't tolerate a missing one. Expand-first ordering means a half-failed run leaves prod on old screens + new (unused) schema — a safe place to stop.

On failure

The migrate step failing fails the whole run — no screens ship. Fix forward with a new migration and redeploy. The history table shows exactly which environment is at which version.
